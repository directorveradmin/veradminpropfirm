# Handoff Step 4 to Step 5

## Step completed
Step 4 of 12: **Rule Engine Core** was completed as a deterministic logic layer rather than a UI build-out.

## What was finalized
- A rule-engine core was added under `src/lib/domain/rules/`.
- The engine now evaluates accounts in a fixed sequence aligned with the canonical evaluation-order doctrine.
- The implementation covers:
  - account integrity validation
  - lifecycle-aware evaluation
  - hard floor logic
  - daily floor logic
  - dominant floor resolution
  - lives and fractional lives
  - progression / minimum-day / target evaluation
  - payout-readiness and payout blockers
  - restrictions
  - dominant mode resolution
  - permissions
  - alert outputs
  - explanation outputs
- A thin service adapter was added in `src/lib/services/accountEvaluationService.ts` so repositories can remain persistence-focused while Step 4 consumes stored facts and produces derived truth.
- Fixture-driven scenario helpers and Node-based Step 4 tests were added without pushing rule semantics into UI code.
- A scenario output printer was added so Step 4 evaluation results can be inspected directly from the demo and edge fixture packs.

## Key decisions made
1. **Engine home**
   - The deterministic core lives under `src/lib/domain/rules/` to preserve the repo boundary defined in Step 2 and Step 3.

2. **Exact-floor behavior**
   - Hard-floor equality is treated as `stopped`, not yet `breached`.
   - Hard-floor values below the floor are treated as `breached`.
   - This preserves distinct behavior for exact-floor, one-cent-above, and one-cent-below scenarios.

3. **Risk unit choice**
   - Because Step 3 profile payloads do not yet contain an explicit risk-unit field, Step 4 uses a standard full risk unit of **1% of starting balance** as the temporary deterministic default.
   - This should become explicit profile configuration later instead of remaining an implicit engine constant.

4. **Tradability floor**
   - Accounts below a minimum effective-lives threshold are treated as non-tradable even if they are technically above the breach floor.
   - This keeps the engine safe-by-default rather than over-permissive.

5. **Payout readiness is lifecycle-aware**
   - Funded accounts can be payout-relevant, but payout readiness becomes fully actionable only when lifecycle and blocker state agree.
   - Pending payout workflow state remains distinct from trading permission.

6. **Consistency handling is transparent**
   - If a consistency-enabled profile lacks enough recent trade data, the engine returns `not_evaluated` instead of inventing a false clean pass.

## Unresolved items
1. **Risk-unit configuration**
   - The rule profile payload should eventually carry an explicit risk-unit definition so the engine no longer relies on the temporary 1%-of-starting-balance convention.

2. **Consistency bundle depth**
   - The current repository evaluation bundle does not yet load recent trade logs. The engine supports them optionally, but Step 5 should decide whether the evaluation bundle should include them as first-class stored facts.

3. **Alert persistence workflow**
   - Step 4 returns alert outputs, but Step 5 should decide how and when derived alerts are persisted, refreshed, or resolved through services.

4. **Simulation path**
   - Step 4 established the live deterministic core. Step 5 can now build the first simulation and consequence-review workflows on top of the same evaluation contract.

## Recommended next-step goal
Step 5 should consume Step 4 engine outputs in a disciplined application layer and begin connecting them to account-detail and workflow surfaces without moving rule semantics into screens.
