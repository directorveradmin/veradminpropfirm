## 1. Step 12 intent

This document protects Veradmin from:
- random feature accumulation,
- AI novelty drift,
- cloud-first pressure that weakens the local-first core,
- and post-v1 clutter that makes the product noisier instead of stronger.

## 2. Frozen decisions

### 2.1 Hardening still outranks novelty
The first post-v1 question is not what flashy thing to add next.
It is what strengthens trust, clarity, protection, and review quality.

### 2.2 Doctrine remains the acceptance gate
A feature is not accepted merely because it sounds useful.
It must strengthen:
- tactical clarity,
- protection,
- explainability,
- business awareness,
- continuity,
- or operator leverage.

### 2.3 Local-first remains the anchor
Optional sync or companion features may exist later.
They do not become the default mental model.

### 2.4 Deterministic truth remains primary
The authoritative layer remains:
- rule engine,
- account evaluation,
- mode assignment,
- tradable/restricted/stopped state,
- payout-readiness logic,
- deterministic simulation,
- derived alerts from governed rules.

### 2.5 AI remains optional, labeled, and bounded
AI may support explanation, summarization, reporting, and guided review.
AI must never silently alter state or outrank deterministic truth.

## 4. AI-safe assistance categories

Safe categories include:
- grounded daily briefings,
- review-summary drafting,
- explanation support,
- reporting assistance.

They must be:
- grounded in deterministic state,
- visibly labeled,
- inspectable against source records.

## 5. AI-prohibited categories

AI must not:
- override rule or state output,
- mutate state silently,
- create predictive trading authority,
- become a chat-first bypass around product surfaces.

## 6. Expansion gate questions

Before accepting a post-v1 item, answer:
1. Does it reduce operator friction meaningfully?
2. Does it improve trust, clarity, protection, planning, or business review?
3. Is it grounded in deterministic state where required?
4. Could the same value be achieved more simply?
5. What docs and tests must change?

## 7. Definition of done

This discipline layer is satisfied when:
1. post-v1 work is layered instead of chaotic,
2. local-first and deterministic truth remain primary,
3. AI is constrained to safe supportive roles,
4. risky ideas are delayed rather than normalized early,
5. roadmap changes stay tied to docs and tests.
